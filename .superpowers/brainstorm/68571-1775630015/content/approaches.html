动效实现方案对比

为 DuanDB 全面添加动效的三种技术路线

A

CSS Transitions + Keyframes（推荐）

思路：基于 Ant Design Motion Token 体系，统一定义 CSS 变量和动画工具类，系统化应用到所有交互场景。

优势

  • 零新依赖，与现有代码完全兼容
  • 性能最优（GPU 硬件加速）
  • 项目中已有部分 CSS 动画基础
  • 与 Ant Design 组件内置动效一致

局限

  • 复杂编排需要额外 JS 辅助
  • Enter/Exit 动画需手动管理 DOM 卸载时机
B

Framer Motion

思路：引入 framer-motion 库，用声明式 API 管理组件动画，AnimatePresence 处理卸载动画。

优势

  • 声明式 API，开发体验好
  • AnimatePresence 完美处理 exit 动画
  • 支持手势、拖拽、布局动画
  • 编排能力强（stagger、orchestrate）

局限

  • 新增 ~30KB 依赖
  • 与 Ant Design 组件动效不统一
  • 团队需学习新 API
  • 部分场景需包装现有组件
C

CSS + rc-motion（Ant Design 内部方案）

思路：使用 Ant Design 内部的 rc-motion 组件（CSSMotion），配合 CSS 类名实现过渡。

优势

  • 与 Ant Design 组件使用相同方案
  • 已作为 antd 间接依赖存在
  • 支持 appear/enter/leave 生命周期

局限

  • API 文档少，社区资源有限
  • 底层 API，开发体验不如 Framer Motion
  • 仍需大量 CSS 编写
